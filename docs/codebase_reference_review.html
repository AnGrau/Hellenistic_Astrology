Architecture

Three packages, each responsible for one kind of work, wired together by two entry points. core never renders anything; docgen and interpretation never calculate anything — the boundary is enforced by convention and checked in review, not by a type system.

The document pipeline

Input birth-data.json name, lat/long or place, local time + tz or UTC datetime
→
core.chart build_observation() Swiss Ephemeris → whole-sign houses, sect, dignities, aspects, Lots, releasing
→
core.observation Observation one frozen dataclass — the sole handoff between calculation and rendering
→
docgen + interpretation .docx + brief.md Phase 1/2 document, plus a factual brief for human-assisted Phase 3 drafting
→
assembly _final.docx appends the reviewed Phase 3 prose — never overwrites the source .docx

Package layering

core — pure calculation docgen / interpretation — rendering entry points

Two entry points orchestrate the same three packages: cli.py (one JSON in, one .docx out) and mcp_server.py (four stdio MCP tools — compute_observation, generate_document, generate_interpretation_brief, assemble_final_document — used by Claude Code, Claude Desktop and Mistral Vibe). Two Claude Skills sit above both: hellenistic-astrology orchestrates the full run end to end, with a hard stop for human review before Phase 3 is ever assembled; hellenistic-astrology-phase3 holds the drafting rules for that step alone.

Data model

Everything downstream of core consumes exactly one object graph. Both dataclasses are frozen — a position, once computed, cannot be quietly mutated by a rendering step.

PointPosition — one placed point (planet, angle, Node, or Lot)

  • name, sign, degree_in_sign, house
  • element, modality, house_quality — universal, all points
  • retrograde, speed — classical planets + Nodes only
  • essential_dignity — domicile/exaltation/exile/fall/peregrine
  • triplicity_dignity, bound_dignity, decan_dignity
  • sect_role — sect light / benefic / malefic / neutral, in or out of sect

Observation — the full factual reading of one chart

  • ascendant, midheaven, planets[7]
  • part_of_fortune, part_of_spirit, part_of_eros
  • north_node, south_node
  • all_points — canonical display order, shared by every table and by cluster/aspect calculation
  • clusters, cluster_aspects — sign groupings and the aspects between them
  • rulerships, mutual_receptions, solar_proximity
  • lunation_phase, eclipse
  • zodiacal_releasing_fortune, zodiacal_releasing_spirit — L1+L2 chapters, 100-year horizon

A recurring pattern worth naming: three modules — dignities.solar_proximity, eclipse.eclipse_configuration, and sect.mercury_is_morning_star's caller — need aspects.angular_gap, but importing it at module scope would create a cycle (aspects already imports PointPosition from observation, which imports dignity/eclipse types back). All three route around it with the same late import inside the function body — a deliberate, repeated convention rather than three separate workarounds.

Code review

Read line-by-line: every module in core/, docgen/, and interpretation/, plus cli.py, mcp_server.py, assembly.py, and bibliography.py — about 3,400 lines. Findings below are what survived; most things that looked odd on first read turned out to be documented, deliberate, and already validated against the two reference charts.

No correctness bugs found. The four items below are minor — duplicate computation, an untested branch, one uncaught exception class, and stale scaffold metadata. Nothing here affects the accuracy of a generated chart.
Sign/degree recomputed up to four times per planet simplification
core/chart.py — build_observation(), lines 76–95

For each of the 7 classical planets, houses.sign_name(raw.longitude) is called separately for essential_dignity, triplicity_dignity, and twice more inside bound_dignity/decan_dignity alongside houses.degree_in_sign(raw.longitude) — then make_point() recomputes both again internally to build the PointPosition. All pure arithmetic (a modulo and a division), so there's no measurable cost, but hoisting sign = houses.sign_name(raw.longitude) once per planet and passing it through would remove the duplication cleanly.

The Swiss Ephemeris file-backed path is never exercised by the test suite test-coverage
core/ephemeris.py — compute_flags()

data/ephe/ is present but empty in this environment (gitignored, populated only by scripts/fetch_ephemeris.py), so compute_flags() always takes the Moshier/Moseph fallback branch (swe.FLG_MOSEPH) — the higher-precision swe.FLG_SWIEPH branch has no test coverage at all, here or presumably in CI. The two are documented as agreeing to ~1 arcsecond, so this is unlikely to hide a real bug, but it means the code path most users would actually run in production (with real ephemeris files installed) is the one path never executed by pytest.

CLI doesn't catch a save failure the same way it catches a calculation failure robustness
cli.py — main(), lines 60–69

build_observation() is wrapped in except (GeocodingError, ValueError) and prints a clean erreur : ... line, but document.save(output_path) just below it is not — a permissions problem, a full disk, or a bad -o path would surface as a raw traceback instead of the friendly message the rest of the CLI uses. Same shape of gap doesn't exist in mcp_server.py, which wraps each tool call uniformly via _with_error_handling.
